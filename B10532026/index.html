xv6系統將使用者可執行的程式(user space)和系統程式(kernel space)分開，兩者之間必須透過kernel提供的system call進行溝通，通常user-level的程式無法直接操作硬體資源，如下圖所示。

其中每個程式被執行、載入到記憶體之後成為一個process，並且擁有自己的PID以供作業系統辨別不同的process；不同process之間彼此是以樹狀結構管理，並且透過兩個很重要的system call— fork() 和 exec() 來實作。

舉例來說，傳統只有文字介面的Linux系統透過一支名為shell的程式來讓使用者對作業系統下達指令，shell本身在開機載入之後也會成為一個process，這時若使用者輸入了例如ls（列出目前資料夾下的檔案）的指令，則shell會呼叫 fork() 、產生一個新的process，此時這個process會接替父輩process的程式碼繼續執行並且共用記憶體空間，因此透過 exec() 替換binary code，達成執行一支新程式的效果，父程式則呼叫 wait() 以等待子程式結束執行後取回執行權。












int pid = fork();
if(pid > 0){
    printf("parent: child=%d\n", pid);
    pid = wait();
    printf("child %d is done\n", pid);
} else if(pid == 0){
    printf("child: exiting\n");
    exit();
} else {
    printf("fork error\n");
}







char *argv[3];
argv[0] = "echo";
argv[1] = "hello";
argv[2] = 0;
exec("/bin/echo", argv);
printf("exec error\n");

xv6還有一個很重要的概念叫做file descriptor，它可以將檔案、I/O和pipe等資源抽象化，讓程式使用時可以有統一的介面；讀取跟寫入都是透過 read() 和 write() 進行包裝，而不需要實際知道自己是在操作什麼對象。

if(fork() == 0) {
    write(1, "hello ", 6);
    exit();
} else {
    wait();
    write(1, "world\n", 6);
}

另外file descriptor還有一個特點是透過父process和子process會共用相同的byte offest（每次操作完檔案後會移動檔案的指標，下一次讀寫會從上一次結束的地方開始）。也就是說上述程式的效果就是父process透過 wait() 保證子process先執行，因此file descriptor為1的檔案會先被子process寫入"hello “，接著父process寫入"world”，結果便是"Hello World"。另外還有一個叫做 dup() 的system call也具有複製file descriptor的功能，讓不同的process可以針對同樣的檔案做讀寫。

pipe是kernel中一組小型的buffer，一端用於寫入、一端用於讀取，方便不同process之間進行溝通。

















int p[2];
char *argv[2];
argv[0] = "wc";
argv[1] = 0;
pipe(p);
if(fork() == 0) {
    close(0);
    dup(p[0]);
    close(p[0]);
    close(p[1]);
    exec("/bin/wc", argv);
} else {
    close(p[0]);
    write(p[1], "hello world\n", 12);
    close(p[1]);
}

上述程式便是建立一組pipe，父process寫入"hello world\n"到寫入端（p[1]），子process將讀取端（p[0]）連結到file descriptor 0然後呼叫 exec() 執行wc計算字數。




echo hello world | wc

echo hello world >/tmp/xyz; wc </tmp/xyz

觀察上述指令，可以發現使用pipe還有許多好處：

  1. pipe會自動清理，而傳統方式使用者必須手動管理暫存檔
  2. pipe可以接收任意長度的資料，反之需要注意儲存空間是否足夠
  3. pipe的兩端可以平行處理，不需要等到前者執行完才能執行後者
  4. 將pipe用於process間的訊息傳遞時，blocking方式較檔案式的非blocking更有效率

最後要介紹的是file system。xv6和Linux的檔案系統也可以視為一組樹狀結構，從系統最底層的root目錄開始，寫成路徑就是類似/a/b/c這樣的型式，代表root目錄下a資料夾中b資料夾之內的c檔案。

有關檔案系統有兩種資料結構需要介紹，分別是stat和inode。透過 fstat() 函數可以取得一組file descriptor所指向檔案（當然由於fd的特性，也可能是資料夾或是硬體裝置）的資訊，將這些資訊填入一組stat結構中。













int fstat(int fd, struct stat *buf);

#define T_DIR 1 // Directory
#define T_FILE 2 // File
#define T_DEV 3 // Device
struct stat {
    short type; // Type of file
    int dev; // File system’s disk device
    uint ino; // Inode number
    short nlink; // Number of links to file
    uint size; // Size of file in bytes
};

inode則是硬碟上實際紀錄一個檔案所屬資訊的方式，而inode和檔案資料是分開存放的；也就是說當我們指定存取一個檔案時，實際上是系統先找到這個檔案名所對應的inode number進而讀取該inode所記錄的檔案資訊，最後才是真的開啟檔案。

這樣的方式也有一個特點，那就是檔案名稱和inode之間的關係稱為link，使用 link() 這個system call可以讓兩個檔案名稱連結到同樣的inode（上面的stat中的nlink變數為2），unlink() 則可以解除一組檔案名稱和inode的對應關係，因此只有當一個inode所對應的link變為0時，該檔案所使用的空間才會真的被清除。





open("a", O_CREATE|O_WRONLY);
link("a", "b");

unlink("a");

Chapter 1

首先，作業系統存在的目的是幫助我們管理各個程式的執行情況；試想若沒有作業系統作為中介，任意程式都可以取得完整的硬體使用權限，則要是程式設計師不懷好意，讓自己的程式強佔所有資源，其他程式就都無法正常運作。

在x86處理器的設計架構中有user mode和kernel mode兩種模式，其中只有處理器處於kernel mode時才能執行諸如檔案讀寫等privileged instructions。也因為一般user mode的process不被直接允許執行這些較為底層的指令，此時作業系統便可以把關、驗證這些system call是否有效，進而阻絕惡意程式並維持系統不受影響。

作業系統依照kernel的規模又可分成以下兩類：

  • monolithic kernel
  • microkernel

在monolithic kernel的設計中，作業系統的所有元件均擁有完整的硬體控制權（執行於kernel mode下），好處是架構簡單、且不同元件之間進行溝通較為便利，但缺點則是system call等interface可能過於複雜，且任何一個部分出錯都會導致整個kernel發生crash，電腦必須重新開機。

反之，將作業系統大部分功能實作於user mode中，kernel只保留最核心部分、盡量輕量化便稱為microkernel。

為了讓每個process都確信自己擁有完整的系統控制權，xv6使用pagle table來在virtual address（process認為自己可以使用的記憶體位置）和physical address（CPU實際可以操作的部分）之間轉換。

如上圖所示，每個process的address space從0開始，由下往上分別是程式資料和global變數、stack和heap；再往上則是kernel的指令和資料，將kernel也一同mapping到address space的好處是，執行system call時可以直接參照個別process的資料。

每個process實際執行的最小單位是thread（一個process可能有好幾個thread），而xv6為每個process建立了一個叫做proc的struct結構用以記錄該process的各種狀態。

比方說，每個process都會有兩個stack：一個user stack和一個kernel stack，如果是執行user space的指令，則只有user stack會被使用；當執行privileged instructions時才會切換回kernel stack，就這樣在兩者之間一直來回。

其他還有state負責記錄這個process目前是處於running、ready、loaded into memory或是waiting for I/O等狀態；pgdir則是該process的page table。
















struct proc {
    uint sz; // Size of process memory (bytes)
    pde_t* pgdir; // Page table
    char *kstack; // Bottom of kernel stack for this process
    enum procstate state; // Process state
    int pid; // Process ID
    struct proc *parent; // Parent process
    struct trapframe *tf; // Trap frame for current syscall
    struct context *context; // swtch() here to run process
    void *chan; // If non−zero, sleeping on chan
    int killed; // If non−zero, have been killed
    struct file *ofile[NOFILE]; // Open files
    struct inode *cwd; // Current directory
    char name[16]; // Process name (debugging)
};

電腦開機之後會先從硬碟將boot loader載入至記憶體，接著boot loader再將kernel從硬碟中讀取出來，並且從entry的位置開始執行。接著為了讓kernel剩餘的部分能夠正常運作，entry會建立一個page table，將虛擬記憶體0x80000000（KERNBASE）map到實體記憶體0x0，最後是將KERNBASE+0x400000 map到0x400000，如此一來會將kernel的指令和資料空間限制在4 Mbytes之內。

系統首先會呼叫userinit成為第一個process，接著userinit又會呼叫allocproc，目的是搜尋process table中有無未使用的空間，有的話就將其標記為已使用並且嘗試為kernel thread分配stack空間還有為該process設定專屬的pid。

xv6中user mode和kernel mode之間的切換是透過interrupt達成的，因此進行切換的時候，系統會將user mode中所使用到的register中的資料存放到kernel stack中，return時就可以把資料取出、繼續運作。

PART 3

通常程式的流程都是讀取指令->更新program counter->讀取指令…一直循環下去，但也有一些情況必須由作業系統（kernel）介入，大致有以下三種：

  1. system call，通常是程式要求執行一些由系統提供的服務
  2. exception，比方說除以0或是存取page table以外的記憶體空間等非法操作
  3. interrupt，例如clock cycle定期引發interrupt以達成time sharing

上述三者的共通點包含系統必須妥善存放目前程式執行中，如暫存器內容等資料以利程式返回時繼續執行、知道system call參數等引發事件的資訊以及kernel執行的開始位置等。另外需要注意的是雖然interrupt和trap這兩個詞會交替使用，但trap是由目前處理器執行的process發出的；interrupt則是由硬體所產生，也可能跟目前執行的process無關。

x86一共支援4種層級的保護機制，由0（權限最高）到3（權限最低），而通常大部分的OS只會使用0跟3，分別對應要kernel mode和user mode。執行中指令的權限等級（current privilege level）會存放在%cs暫存器中的CPL欄位。

interrrupt descriptor table（IDT）則記錄了在不同的interrupt情況下，%cs和%eip暫存器該如何操作。

發生system call時，程式會執行 int 指令，過程描述如下：

  • 從IDT中取得第n筆descriptor，n是int執行的參數
  • 檢查%cs中的CPL是否<=DPL，DPL是前述取得descriptor的privilege level
  • 若DPL<CPL則將%esp和%ss的值儲存在CPU的內部暫存器中
  • 從前述的descriptor中讀取%ss和%esp
  • push %ss
  • push %esp
  • push %eflags
  • push %cs
  • push %eip
  • 若是interrupt的話則清除%eflags中的IF bit
  • 將%cs和%eip設為descriptor中的值

在 int 執行完之後，系統會使用 iret 來達成return的目的，它會將 int 過程中存進stack的值pop出來，然後恢復%eip的值讓程式依照原本的順序繼續執行。

Code: Assembly trap handlers

x86總共允許256種不同的interrupt，其中0-31是軟體的interrupt（除以0或是存取非法記憶體位置等）、32-63是硬體的interrupt，最後64定義為system call。









void tvinit(void)
{
    int i;
    for(i = 0; i < 256; i++)
        SETGATE(idt[i], 0, SEG_KCODE<<3, vectors[i], 0);
    SETGATE(idt[T_SYSCALL], 1, SEG_KCODE<<3, vectors[T_SYSCALL], DPL_USER);
    initlock(&tickslock, "time");
}

當interrupt發生時，系統會執行如下操作：

處理器處於user mode時，從task segment descriptor讀取%esp和%ss，將原本的%esp和%ss；若是處於kernel mode，則以上操作皆不發生。接著push %eflags、%cs和%eip，對於page fault等種類的interrupt則連錯誤資訊一並存入，最後從IDT讀取%eip和%cs。

剩下的工作會交給 Alltraps 完成；push %ds、%es、%fs和%gs，以及其他的通用暫存器。這時kernel stack就會有一個完整的 trapframe 結構，好讓系統在執行權從kernel交回給原本的process時，能保有和當初相同的工作狀態。會儲存%eax則是為了記錄稍後要執行的system call編號。
































struct trapframe 
{
    uint edi;
    uint esi;
    uint ebp;
    uint oesp;
    uint ebx;
    uint edx;
    uint ecx;
    uint eax;

    ushort gs;
    ushort padding1;
    ushort fs;
    ushort padding2;
    ushort es;
    ushort padding3;
    ushort ds;
    ushort padding4;
    uint trapno;

    uint err;
    uint eip; //int指令後下一個指令的位置
    ushort cs; //user code segment selector（指向程式碼的大區塊）
    ushort padding5;
    uint eflags; //記錄處理器目前的狀態

    uint esp;
    ushort ss;
    ushort padding6;
};

Code: System calls

有了上述知識，最後我們還必須知道system call的參數，argint、argptr、argstr 和 argfd這幾個函式分別用於取得int、pointer、string和file descripot類型的參數。其中%esp會指向system call的return address，參數剛好就在%esp+4的地方，因此第n筆參數的位置就是%esp+4+4*n。

Code: Interrupts

interrupt某些部份和system call很類似，但不同之處是system call通常是根據固定的程式邏輯發出的，但interrrupt有可能是由硬體裝置在任何時間所產生。舉個例子，主機板上的timer裝置每100ms發出一次interrupt，讓系統可以記錄時間和執行time sharing，但這樣的頻率又不會高到過於干擾系統運作。

timer訊號的傳遞分成兩個部份，I/O裝置（the IO APIC, ioapic.c）和處理器內部（the local
APIC, lapic.c）。IO APIC有一張table讓處理器可以記錄什麼樣的I/O裝置對應到哪個欄位。不同的裝置會觸發不同種類的interrupt，並且可以指定要將這個interrupt訊號傳遞到哪一個處理器，比方說xv6將鍵盤事件傳送到0號處理器，硬碟事件則傳送到編號最大的處理器。



lapicw(TIMER, PERIODIC | (T_IRQ0 + IRQ_TIMER)); //讓LAPIC週期性地發出interrupt到IRQ_TIMER
lapicw(TPR, 0);//啟動CPU上的LAPIC讓訊號可以傳遞

此外processor也可以決定自己是否要透過%eflags暫存器中的IF欄位來接收interrupt，cli 透過清空IF來關閉；sti 則可以開啟。xv6會在系統開機時關閉各CPU的interrupt功能，scheduler開始執行後又會再度開啟。

Code: Disk driver

kernel在系統開機的時候透過呼叫 ideinit 來初始化disk driver（對IDE硬碟而言），接著ideinit會再呼叫 ioapicenable 來發出IDE_IRQ的interrupt訊號。

接著ideinit會開始檢查硬體狀態，它會先呼叫 idewait 讓硬碟準備好接收指令。接著主機板會將硬碟的status bit呈現在0x1f7的I/O腳位上，然後ideinit會在busy bit (IDE_BSY)被清空和ready bit (IDE_DRDY)被設定好之後接收前述的status bit。

iderw 會根據buf結構中flags的值來決定該對硬碟採取何種讀寫策略：B_DIRTY被設為1的話代表要將buffer的內容寫入硬碟；B_VALID是0的話則將資料從硬碟讀進buffer。

硬碟的讀寫相較CPU而言是很花時間的，同理，一直查看硬碟的讀寫狀態也很浪費時間和資源；因此iderw採用的策略是讓其他process享有CPU的資源、等到讀寫完成等事件發生再透過interrupt通知CPU，這時候需要使用一個queue的結構來記錄那些硬碟操作還在等待中。

iderw會把新進入的buffer資料加入queue的尾端，如果一個buffer已經排到queue的最前頭了則會呼叫 idestart 來進行實際的硬碟操作，否則要等到前面的buffer都已經被處理完了才會輪到自己。










// Append b to idequeue.
b−>qnext = 0;
for(pp=&idequeue; *pp; pp=&(*pp)−>qnext);
*pp = b;

// Start disk on next buf in queue.
if(idequeue != 0)
    idestart(idequeue);
release(&idelock);

將讀寫的request加入queue之後iderw就暫時沒事可做了、必須等待結果揭曉。因此iderw此時會開始進入睡眠狀態，等待操作結束之後interrupt handler修改flags的值。這時CPU就可以將資源分配給其他process以免浪費運算資源。





// Wait for request to finish.
while((b−>flags & (B_VALID|B_DIRTY)) != B_VALID){
    sleep(b, &idelock);
}

最後讀寫終於完成、並且觸發interrupt之後，trap會呼叫 ideintr 接手剩餘的工作。此時ideintr會詢問queue裡的第一個buffer，藉以得知方才完成的操作是讀還是寫，如果是讀取、disk controller中也有資料正在等待的化，ideintr會使用 insl 將資料讀進buffer；最後設定B_VALID、清空B_DIRTY、喚醒其他正在等待的process，最後把下一個等待中的buffer交給disk。














// Read data if needed.
if(!(b−>flags & B_DIRTY) && idewait(1) >= 0)
    insl(0x1f0, b−>data, BSIZE/4);
    
// Wake process waiting for this buf.
b−>flags |= B_VALID;
b−>flags &= ~B_DIRTY;
wakeup(b);

// Start disk on next buf in queue.
if(idequeue != 0)
     idestart(idequeue);
release(&idelock);
